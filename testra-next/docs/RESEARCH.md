# 調査: TESTRA / QualityForward と上位互換戦略

> 目的: 本プロジェクト（TESTRA-Next）が「何を上位互換とするか」を定義するための調査メモ。
> 出典は一次情報が取れる範囲で記載し、未確認の推定は「推定」と明示する。

## TESTRA（ベリサーブ）

- 位置づけ: 「人×AIで品質保証する**テスト設計エージェント**」。
  仕様に対しテスト観点を適用し、テスト設計をAIが支援する（本リポジトリ `QA-PMO` の既存記述と整合）。
- 中心スコープ: **テスト設計〜テストケース生成**。
- 特徴: 観点ベースの設計支援、人のレビューを前提とした協働型。

## QualityForward（ベリサーブ）

- 位置づけ: クラウド型**テスト管理**サービス。テスト計画・テストケース・実行結果・進捗を一元管理。
- 主要概念（マッピング対象）: Project / Test Suite / Test Case / Test Run(Result)。
- 本プロジェクトは `connectors/qualityforward.js` で
  「フィーチャー→Suite / ローケース→Case / 実行結果→Result」を対応づける。

> 注: API仕様の細部（エンドポイント/認証ヘッダ名）は環境により異なるため、
> コネクタは `baseUrl`/`apiKey`/`fetchImpl` を注入可能にし、実接続は別環境で検証する設計とした。

## 「圧倒的上位互換」の定義

TESTRA が強い「テスト設計」を包含しつつ、その**前後の工程まで自動で貫通**させることを上位互換の核とする。

| 軸 | TESTRA | TESTRA-Next の上位互換ポイント |
|---|---|---|
| 工程範囲 | 設計中心 | 取込→分析→設計→ケース→**スクリプト→実行→連携→レポート** |
| 入力多様性 | 仕様テキスト | **URL・apkメタ**も一次入力に |
| 技法適用 | 観点適用（人依存大） | 29119-4 技法をシグナルで**自動選定**＋最低保証 |
| 成果の実行性 | 設計止まり | **実行可能スクリプト**（Gherkin/Playwright/Appium）まで生成 |
| 管理連携 | 個別 | **QualityForward へ自動同期**（dry-run既定で安全） |
| 実行形態 | SaaS | **オンプレ／クラウド／CLI** 同一コア |
| 再現性/監査 | — | 決定論ID＋要件↔結果トレーサビリティ |

## 設計上の割り切り（正直な現状）

- 形態素解析は未使用。要件抽出・フィーチャー命名はキーワード/句読点ベースのため、
  日本語の砕けた文ではフィーチャー粒度がぶれる（→ `GOALS.md` で改善予定）。
- apk/docx/pdf の**バイナリ実解析は未接続**。現状はメタ供給を受ける契約のみ。
- 実行ステージは既定シミュレーション。実ブラウザ/実機実行は runner 注入で拡張。

## 準拠規格

- ISTQB（severity 分類：Critical/High/Medium/Low）
- ISO/IEC/IEEE 29119-4（テスト設計技法）
- ISO/IEC 25010（品質特性：非機能フィーチャー分類）
