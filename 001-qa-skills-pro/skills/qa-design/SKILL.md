# /qa-design — 規格準拠テスト設計

仕様またはコードを入力とし、テスト設計技法を規格どおりに適用してテストケースを導出する。
我流の「思いつきテスト」は生成しない。全ケースに根拠、レポート末尾にカバレッジ宣言を付ける。

## 手順

### 1. 共通ポリシーの読み込み

`../qa-shared/evidence-first-policy.md` と `../qa-shared/report-format.md` を読む。
以降の全出力はこの2ファイルに従う。

### 2. テストベースの特定

- 入力が仕様書なら: テスト対象の要求項目を列挙し、各項目に識別子を振る（既存のREQ-ID等があればそれを使う）。
- 入力がコードなら: 公開インターフェース（関数シグネチャ・API・画面入力）を列挙する。**実装の内部ロジックからテストを導出しない**（実装をなぞるトートロジーテストの禁止。期待値は仕様・ドキュメント・型制約から導出する）。
- テストベースが曖昧・欠落している場合は、不足箇所を「要人間確認リスト」に挙げた上で、確認できた範囲だけを設計対象とする。

### 3. 技法の選択

対象の入力構造を分析し、該当する技法の参照ファイルを読み込む。複数該当は併用する。

| 入力構造 | 技法 | 参照ファイル |
|---|---|---|
| 値域・分類のある入力（数値範囲、列挙、文字種） | 同値分割 | `references/equivalence-partitioning.md` |
| 順序尺度の境界を持つ入力（範囲、桁数、件数、日付） | 境界値分析 | `references/boundary-value-analysis.md` |
| 複数条件の組み合わせで結果が決まるビジネスルール | デシジョンテーブル | `references/decision-table.md` |

初版でカバーしない構造（状態遷移・組み合わせ爆発する多パラメータ等）を検知した場合は、
本スイート初版の対象外技法（状態遷移テスト／ペアワイズ法）が適することを明記して
手動設計を推奨する。無理に3技法へ当てはめない。

### 4. 技法の適用

各参照ファイルの手順に厳密に従う。中間成果物（パーティション表・境界一覧・デシジョンテーブル本体）を
省略せずレポートに含める。中間成果物がないテストケースは検収時に検証できない。

### 5. 出力

`../qa-shared/report-format.md` のテストケース構造で出力し、最後に
`references/coverage-declaration.md` の書式でカバレッジ宣言を付ける。

## 禁止事項

- 根拠（仕様該当箇所）のないテストケースの出力
- 実装コードの現在の挙動を期待値としてコピーすること（バグの固定化）
- カバレッジ宣言の省略・「網羅した」等の定性的宣言での代替
- パーティション表・デシジョンテーブル等の中間成果物の省略
