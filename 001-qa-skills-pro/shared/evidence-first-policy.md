# Evidence-First ポリシー（全QA Skill共通）

判定はevidence-only。根拠を提示できない指摘・テストケースは**出力しない**。
このポリシーはqa-design / qa-review / qa-nonfuncの全出力に適用される。

## 根拠として認めるもの

| 種別 | 形式 | 例 |
|---|---|---|
| 仕様該当箇所 | ファイルパス＋見出し or 行番号＋原文引用 | `spec.md「3.2 割引計算」:「会員は10%引き」` |
| コード位置 | `path:line` ＋該当コードの引用 | `src/discount.ts:42 (rate = isMember ? 0.1 : 0)` |
| 実行コマンドと結果 | 実際に実行したコマンドと出力の要約 | `npm test -- discount → 2 failed` |
| 再現手順 | 番号付きの操作列＋期待/実際の差 | 手順1〜4＋「期待: エラー表示／実際: 無応答」 |
| 規格・技法の定義 | 規格名＋項目名 | `ISO/IEC 25010 信頼性—障害許容性` |

## 根拠として認めないもの

- 「一般的に〜すべき」「〜が望ましい」だけの一般論（仕様・コードのどこが該当するか示せていない）
- 読んでいないファイルへの推測（「おそらく〜になっている」）
- 実行していないコマンドの結果予測
- 仕様に存在しない要求の捏造（要求の不在を指摘する場合は「spec.mdに該当記述なし」という不在確認自体を根拠として明記する）

## 出力前セルフチェック（必須）

出力を確定する前に、全指摘・全テストケースについて以下を確認する:

1. `evidence` フィールドが埋まっているか → 空なら**その項目を削除**する
2. 引用した仕様箇所・コード位置は実在するか → 実際にファイルを開いて確認していないものは削除する
3. 推定を含む場合、`confidence: 推定` を明示し、確認方法を併記しているか

## 推定の扱い

コードや仕様を確認できない箇所についてどうしても言及が必要な場合のみ、
`confidence: 推定`（要人間確認）とラベル付けし、「何を確認すれば確定するか」を必ず併記する。
ラベルなしの推定は禁止。
