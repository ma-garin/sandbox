# Part 1: UI/UX 基礎原理

## この章で学ぶこと

UI/UX を理解するための基礎用語・概念・フレームワーク。「UX とは何か」から始まり、認知・視覚・操作の原理まで体系的に整理する。

## 対象者

- UI/UX を初めて学ぶ人
- 用語の定義を正確に押さえたい開発者・QAエンジニア

---

## 1-1. UX / UI / CX / Service Design の違い

| 用語 | 定義 | スコープ |
|------|------|--------|
| UI (User Interface) | ユーザーが直接触れる画面・操作部分 | 画面単位 |
| UX (User Experience) | 使用前〜使用後を含む体験全体 | ジャーニー全体 |
| CX (Customer Experience) | ブランド接点すべてを含む顧客体験 | 企業全体 |
| Service Design | サービスを構成する人・プロセス・物を設計する | 組織・社会まで |

---

## 1-2. UX を構成する 5 層（Garrett の Elements of UX）

```
表層: Visual / UI / Copy
  ↑
骨格: Layout / Interaction Design
  ↑
構造: IA / Navigation / User Flow
  ↑
要件: User Needs / Business Goals
  ↑
戦略: Problem / Context / Value Proposition
```

> **ポイント**: 表層の問題は多くの場合、構造・要件・戦略レベルの問題が表れたもの。表層だけ直しても根治しない。

## 1-3. 人間中心設計（HCD）

ISO 9241-210 に基づくプロセス。

```
利用状況の把握 → ユーザーと要求事項の明確化 → 設計案の作成 → 評価
                        ↑_____________フィードバック________________|
```

### HCDチェックリスト

- [ ] 利用状況（環境・端末・ネットワーク・ユーザー属性）を定義したか
- [ ] ユーザーの目標とタスクを分析したか
- [ ] ユーザビリティ基準を設定したか
- [ ] プロトタイプでユーザーテストを実施したか

## 1-4. 認知心理の基礎

| 概念 | 説明 | UI への影響 |
|------|------|-----------|
| 注意 | 同時に意識できる対象は限られる | 重要情報を目立たせる、不要な要素を省く |
| ワーキングメモリ | 短期的に保持できる情報は 7±2 チャンク | Miller の法則。選択肢を絞る |
| メンタルモデル | ユーザーが持つ「こう動くはず」という期待 | Jakob の法則。既存パターンを踏襲する |
| 認知負荷 | 情報処理に必要なコスト | シンプルに保つ。段階的に開示する |

---

## 1-5. 視覚設計の原則（ゲシュタルト法則）

| 法則 | 内容 | UI例 |
|------|------|------|
| 近接 | 近いものはグループとして認識される | フォームラベルと入力欄を近づける |
| 類同 | 似たものは同じグループと認識される | 同種ボタンは同じ色・形にする |
| 連続 | 連続した要素はひとまとまりに見える | ステッパー・プログレスバー |
| 閉合 | 不完全でも完全な形として認識する | アイコン・ロゴデザイン |
| 図と地 | 前景と背景を分けて認識する | モーダル・ドロップダウン |

---

## 1-6. 操作性の基礎

| 概念 | 説明 |
|------|------|
| アフォーダンス | 見た目から操作可能性が伝わること（ボタンは押せそうに見える） |
| フィードバック | 操作結果をユーザーに即時伝える（クリック後の状態変化） |
| エラー回復 | 誤操作を元に戻せる・修正しやすい設計 |
| 一貫性 | 同じ操作・同じ結果。学習コストを下げる |

---

## QA 観点への変換

| UX基礎概念 | QAチェック項目 |
|-----------|--------------|
| 認知負荷 | 初見で次の操作が分かるか |
| フィードバック | 成功・失敗・処理中が明確に表示されるか |
| エラー回復 | 誤操作後に元に戻せるか。エラーメッセージは具体的か |
| 一貫性 | 同じ要素が画面をまたいで統一されているか |
