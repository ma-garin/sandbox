# Part 4: UXリサーチ・評価・計測

## この章で学ぶこと

UX を「感覚」ではなく「データ」で語れるようにする。定性・定量・専門家評価の3軸を整理し、QAエンジニアが参加できる評価手法に落とし込む。

---

## 4-1. UX評価の3軸

| 軸 | 手法 | 得られるもの | 主な担当 |
|----|------|------------|---------|
| 定性 | インタビュー / ユーザビリティテスト / コンテキスト調査 | なぜ・どう感じるか | UXリサーチャー |
| 定量 | ログ分析 / A/Bテスト / SUS | 何人が・どれだけ | アナリスト / エンジニア |
| 専門家評価 | ヒューリスティック評価 / アクセシビリティ監査 | 原則に対する適合性 | UXデザイナー / QAエンジニア |

---

## 4-2. 定性調査

### ユーザビリティテスト

```
目的: ユーザーが実際に操作する様子を観察し、詰まりポイントを発見する
参加者: 5〜8名（主要な問題の85%を発見できる）
形式: 発話思考法（Thinking Aloud）
記録: 画面録画 + アイトラッキング（あれば）
```

**タスク設計の原則**

- シナリオ形式で渡す（「〇〇をしてください」）
- 誘導しない（「〜は分かりましたか？」は NG）
- 5分以内で完了できるタスクに絞る

### インタビュー

| 種類 | 用途 |
|------|------|
| コンテキスチュアルインクワイアリー | 実際の環境で観察しながら質問 |
| 半構造化インタビュー | 事前に質問骨格を作り、深掘りを許容 |
| ダイアリースタディ | 数日〜数週間、日記形式で記録してもらう |

---

## 4-3. 定量計測

### 主要 UX メトリクス

| 指標 | 計算方法 | 意味 |
|------|----------|------|
| タスク成功率 | 成功数 / 試行数 | 操作完了しやすさ |
| タスク完了時間 | 開始〜完了の秒数 | 効率性 |
| エラー率 | エラー発生数 / 操作数 | 設計の安全性 |
| SUS スコア | 10設問の計算式 | 全体的な使いやすさ（68点以上が合格目安） |

### SUS（System Usability Scale）スコアリング

```
10問の設問（5段階評価）に回答
→ 奇数問: スコア - 1
→ 偶数問: 5 - スコア
→ 合計 × 2.5 = SUS スコア（0〜100）

68以上: 平均以上
80以上: 優良
90以上: 最優秀
```

### ファネル分析

```
ページA → ページB → ページC → 完了
  100        80        60       40

離脱率: A→B 20%, B→C 25%, C→完了 33%
→ B→C の離脱が最大の問題箇所
```

### A/B テスト設計

| 要素 | 内容 |
|------|------|
| 仮説 | 「〜を変えると〜が改善する」を1つに絞る |
| 指標 | 主指標1つ + 副指標2〜3つ |
| サンプルサイズ | 統計的有意差が出る人数を事前計算 |
| 期間 | 最低1週間（曜日効果を除外） |

## 4-4. 専門家評価

### ヒューリスティック評価（HE）

```
手順:
1. 評価者 3〜5名を用意
2. Nielsen の 10 ヒューリスティックスを基準に独立評価
3. 個別に問題をリストアップ
4. 評価者を集めてデブリーフィング
5. 重大度スコアを付与してソート
```

**重大度スコア**

| スコア | 意味 |
|--------|------|
| 0 | 問題なし |
| 1 | 小さな問題（優先度低） |
| 2 | マイナー問題（修正が望ましい） |
| 3 | メジャー問題（優先的に修正） |
| 4 | 致命的問題（リリース前に必ず修正） |

---

## 4-5. QAエンジニアが参加できる評価手法

| 手法 | QA が担えること |
|------|--------------|
| ヒューリスティック評価 | Nielsen の 10 原則でレビュー票を作成・評価 |
| 探索的テスト | 迷い・違和感・誤操作を観察してセッションシートに記録 |
| アクセシビリティ監査 | axe / WAVE 等のツールで自動チェック |
| A/B テスト検証 | 測定基準の設計とデータ収集の支援 |
