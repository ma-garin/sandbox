> 版: 1.0 ／ 2026-07-21 ／ ma-garin

## 1. システム概要

多様な拡張子の社内ナレッジを、**構造を壊さずに抽出**し、**AIが読める形に再構成**し、**残る問題を診断**して、RAG投入形式で出力するローカル完結型システム。

```
[社内ナレッジ]        [AI2DocFrendry]                      [出力]
 docx/xlsx/pptx ─┐   ①抽出・構造保持 ★最重要        ┌─ Markdown（構造保持）
 pdf/txt/md/csv ─┼─→ ②再構成 → ③改変検証 ─────────┼─ chunks.jsonl（RAG投入用）
 html/eml       ─┘   ④品質診断                      ├─ metadata.csv
                                                      ├─ diff.md（全変更の差分）
                                                      └─ REPORT.md
```

## 2. 利用者

| ID | 利用者 | 関心 |
|---|---|---|
| U-1 | ナレッジ管理者 | 一括変換したい。何が変換でき、何ができなかったか知りたい |
| U-2 | RAG構築担当 | そのまま投入できるデータが欲しい。品質を確認したい |
| U-3 | 文書作成者 | 自分の文書がAIに読まれるか、どう直すか知りたい |

## 3. 業務要求

| ID | 要求 | 対応課題 |
|---|---|---|
| BR-01 | 多形式を形式を意識せず一括投入できる | P-01 |
| BR-02 | **表・見出し階層・読み順を壊さずに抽出する** | P-01, P-03 |
| BR-03 | **意味を持つ書式の消失を検出する** | P-02 |
| BR-04 | 中身をAIが読める形に再構成する | P-04, P-05 |
| BR-05 | 再構成が原文を歪めないことを保証する | — |
| BR-06 | 投入前に品質を判定し、改善を促す | P-01〜P-06 |
| BR-07 | 機密・陳腐化・重複を検知する | P-06 |
| BR-08 | 外部へ送信しない／コスト0円 | 情報管理規程・予算 |

## 4. 機能要件

> 優先度: **Must**（MVP必須）／**Should**／**Could**

### 4-1. 取り込み

| ID | 要件 | 優先 |
|---|---|---|
| FR-01 | フォルダ（再帰）／ZIP／個別ファイルを投入できる | Must |
| FR-02 | 対象拡張子・除外パターンを設定できる | Must |
| FR-03 | 対応外・破損は処理を止めず隔離し、理由付きで一覧化する | Must |
| FR-04 | 内容ハッシュ一致なら再処理をスキップする | Should |

### 4-2. 抽出・構造保持（★最重要 / ADR-02）

| ID | 要件 | 優先 |
|---|---|---|
| FR-05 | `.docx` `.xlsx` `.pptx` `.pdf` `.txt` `.md` `.csv` `.html` から抽出できる | Must |
| FR-06 | **見出し階層・箇条書き・表・読み順を構造情報として保持する** | Must |
| FR-07 | **表の結合セルを展開し、行列対応を崩さない** | Must |
| FR-08 | **意味を持つ書式（打ち消し線、強調、色分け、テキストボックス）を検出し記録する。消失する場合は警告する** | Must |
| FR-09 | 画像・チャートは本文から除外し、位置と件数を記録する | Must |
| FR-10 | **パーサをプラグイン構造とし、文書タイプ別に切り替えられる** | Must |
| FR-11 | **抽出率（抽出文字数／推定情報量）を算出する** | Must |
| FR-12 | `.eml` `.msg` から件名・日時・本文・添付一覧を抽出できる | Should |
| FR-13 | OCRで画像内文字を抽出できる | Could |
| FR-14 | 旧形式（`.doc` `.xls` `.ppt`）を取り込める | Could |

### 4-3. 正規化・構造化

| ID | 要件 | 優先 |
|---|---|---|
| FR-15 | 文字コードを判定しUTF-8へ統一する（不能時は隔離） | Must |
| FR-16 | 反復ノイズ（ヘッダ・フッタ・ページ番号・目次）を除去し、除去内容を記録する | Must |
| FR-17 | 構造をMarkdownへ再構成する（見出し`#`、箇条書き`-`、表はMarkdown表） | Must |
| FR-18 | **結合セル・階層ヘッダを含む表はHTMLを併記する**（Markdownは構造的に表現不可 / ADR-04） | Must |
| FR-19 | **無関係なマークアップを持ち込まない**（フォーマットノイズは表・数式で−18〜19pt） | Must |
| FR-20 | YAML front matter でメタデータを付与する | Must |

### 4-4. 再構成（層1: ルールベース）

| ID | 要件 | 優先 |
|---|---|---|
| FR-21 | 表記ゆれ（全半角・長音・送り仮名）を統一する | Must |
| FR-22 | 章番号参照を実体化する（「上記2項」→「2. 移行手順」） | Must |
| FR-23 | メタ情報（タイトル・日付・版・所管）をファイル属性等から補完する | Must |
| FR-24 | 略語の初出に定義を付与する（文書内に定義がある場合） | Should |
| FR-25 | 解決できない外部参照（「別紙3参照」）を警告として残す | Must |

### 4-5. 再構成（層2: ローカルLLM・任意）

| ID | 要件 | 優先 |
|---|---|---|
| FR-26 | 見出しのない文書に見出しを生成する | Should |
| FR-27 | 指示語（「標記の件」「上記の通り」）を実体化する | Should |
| FR-28 | 層2を無効化してもシステムが動作する（層1のみで完結） | Must |
| FR-29 | 層2はローカル実行とし、外部送信しない | Must |

### 4-6. 改変検証（層3・決定的）

| ID | 要件 | 優先 |
|---|---|---|
| FR-30 | **原文にない固有名詞・数値・日付が出現していないか照合する**（情報追加の禁止） | Must |
| FR-31 | **原文の各文が再構成後に対応を持つか照合する**（情報欠落の禁止） | Must |
| FR-32 | 数値トークンの多重集合が一致することを確認する | Must |
| FR-33 | 検証を通らない変更は適用せず原文のまま残す（ADR-10） | Must |
| FR-34 | 全変更を種別・方式（ルール／LLM）・信頼度付きで差分出力する | Must |
| FR-35 | 原本を保持し、上書きしない | Must |

### 4-7. チャンク分割

| ID | 要件 | 優先 |
|---|---|---|
| FR-36 | **見出し境界で分割する。semantic chunking は行わない**（ADR-05） | Must |
| FR-37 | 上限トークン超過時のみ段落境界で二次分割する | Must |
| FR-38 | **オーバーラップを文書種別で切り替える**（表を含む=25%、含まない=極小 / ADR-06） | Must |
| FR-39 | **表は行単位でチャンクし、各行の先頭にヘッダを反復する**（ADR-07） | Must |
| FR-40 | **各チャンクの先頭に見出しパスと出典を前置する**（Contextual Retrieval: 失敗率−35%） | Must |
| FR-41 | `chunk_id` / `parent_doc_id` / 見出しパス / 出典を付与する | Must |
| FR-42 | チャンクサイズ・オーバーラップを設定できる | Must |

### 4-8. 品質診断（ICIS 2025 のDQ次元 + OHRBench の2分類 / ADR-11）

| ID | 要件 | 優先 |
|---|---|---|
| FR-43 | **Semantic Noise**（情報の欠落・誤り）と **Formatting Noise**（書式の不統一）を分けて評価する | Must |
| FR-44 | 5次元で採点する — `Intrinsic`（抽出正確性・矛盾）／`Contextual`（重複・陳腐化・網羅）／`Representational`（構造の機械可読性・表記一貫性・略語）／`Accessibility`（抽出可否・参照解決可否）／`Accountability`（機密・所管） | Must |
| FR-45 | **Semantic Noise を Formatting Noise より重く配点する**（前者は逓減しない、後者は逓減する） | Must |
| FR-46 | 総合判定 A〜D を返す（投入可／投入可・軽微／要改善／投入非推奨） | Must |
| FR-47 | 指摘はISTQB severity（Critical/High/Medium/Low）で分類する | Must |
| FR-48 | **全指摘に根拠（引用と位置）を添える。根拠のない指摘を出さない** | Must |
| FR-49 | 指摘ごとに具体的な改善アクションを提示する | Must |
| FR-50 | **再構成前と再構成後の両方を採点し、改善量を示す**（ADR-09相当） | Must |
| FR-51 | 採点は決定的である（同一入力→同一出力） | Must |
| FR-52 | **CER/WER を指標に使わない**（ADR-12） | Must |
| FR-53 | 次元別の重み・しきい値を設定で変更できる | Should |

|---|---|---|

| FR-55 | 検出パターンを設定で追加・除外できる | Must |
| FR-56 | 最終更新から一定期間（既定3年）超過を陳腐化候補として警告する | Must |
| FR-57 | 内容ハッシュ一致による完全重複を検出する | Should |
| FR-58 | 機密情報をマスクした出力を生成できる | Could |

| FR-60 | 一式をZIPで一括取得できる | Must |
| FR-61 | 変換前後をUI上で並べて確認できる | Should |

## 5. 非機能要件（ISO/IEC 25010）

| ID | 特性 | 要件 | 受入基準 |
|---|---|---|---|
| NFR-01 | 機能適合性 | 主要8形式で本文と表を欠落なく抽出する | サンプル各形式3件で抽出率95%以上 |
| NFR-02 | 機能適合性 | **意味を持つ書式の消失を検出する** | 打ち消し線・強調を含むテストファイルで100%検出 |
| NFR-03 | 性能効率性 | 1,000ファイル（平均500KB）を30分以内。層2有効時は60分以内 | 実測 |
| NFR-04 | 性能効率性 | メモリ2GB以内（逐次処理・全件展開の禁止） | 実測 |
| NFR-05 | 互換性 | macOS（M1）と Windows 11 で動作する | 両環境でスモークテスト |
| NFR-06 | 使用性 | 投入から出力まで3ステップ以内 | 操作実測 |
| NFR-07 | 使用性 | 判定と改善アクションを専門知識なしに理解できる | U-3による受入 |
| NFR-08 | 信頼性 | 1ファイルの失敗が全体を停止させない | 破損混入テスト |
| NFR-09 | 信頼性 | 層1・層3は完全に決定的。層2はseed固定で再現性を確保し、差分を必ず提示する | 同一入力2回の出力比較 |
| NFR-10 | **セキュリティ** | 既定構成で外部通信を一切行わない（層2もローカル実行） | 通信監視で送信ゼロ（**最重要**） |
| NFR-11 | セキュリティ | ログ・エラーに本文および機密情報を出力しない | ログ検査 |
| NFR-12 | セキュリティ | 認証情報をソースへ埋め込まない | 静的検査 |
| NFR-13 | 保守性 | **パーサ・再構成操作をプラグイン構造とし、追加を新規ファイルのみで行える** | 追加時の差分確認 |
| NFR-14 | 保守性 | 1ファイル400行以内、関数50行以内、純粋関数 | レビュー |
| NFR-15 | 保守性 | 単体テストカバレッジ80%以上（診断・検証は必須） | 計測 |
| NFR-16 | 移植性 | `venv` と `requirements.txt` のみで構築完了 | クリーン環境で確認 |

## 7. 受入基準

| ID | 基準 |
|---|---|
| AC-01 | 実ナレッジ100件が「変換完了」または「理由付き隔離」に分類される |
| AC-02 | 変換後の見出し・表が原本と目視一致する（サンプル20件） |
| AC-03 | **結合セルを含む表で行列対応が崩れない** |
| AC-04 | **打ち消し線を含む文書で、書式消失がCritical指摘として検出される** |
| AC-05 | 低品質文書（ベタ書き・別紙参照・スキャンPDF）がいずれもC判定以下になる |
| AC-06 | 機密情報を埋めたテストファイルがCritical検出される |
| AC-07 | **再構成後に原文にない固有名詞・数値が出現しない**（層3の検証） |
| AC-08 | 実行中に外部通信が発生しない |
| AC-09 | 出力JSONLがスキーマ検証を通過する |
