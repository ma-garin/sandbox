# プロダクト戦略 — デモの先にある「勝てる」プロダクトへ

本書は、上司が求める「提供メニューのデモ」を**受動的に満たすだけで終わらせず**、
ベリサーブ/SHIFT型の競合に対し優位性を持つプロダクトへ発展させる道筋を示す。

## 1. 中核テーゼ：堀は「技法」ではなく「観点という知識資産」

市販OSS（PICT/axe-core/textlint等）は誰でも使える。技法そのものに差別化はない。
ベリサーブ/SHIFTの本当の優位性は、長年蓄積した**テスト観点・欠陥知識**である。

→ よって我々の防御可能資産も**観点ライブラリ（test viewpoint knowledge base）**に置く。
今回 `js/viewpoints.js` として資産化し、観点ベース設計エンジンに組み込んだ。これは：
- **拡張可能**：業界別・顧客別の観点を追記して“育てる”
- **監査可能**：各テスト条件が 観点→技法→カテゴリ に追跡できる（第三者検証グレード）
- **カバレッジ可視化**：観点の抜け（例：性能/同時実行）を自動警告 = 人手レビューの価値を機械化
- **AIレディ**：この観点DBがそのまま将来のRAG知識源（AI版の精度の堀）になる

## 2. なぜこれが「人月を削る」か

SHIFT/ベリサーブのコスト構造＝**エンジニアの人月**。テスト設計・観点出しは熟練者の時間を最も食う。
観点ライブラリ駆動の設計は、ジュニアでもシニア相当の観点網羅を出せる → **設計工数とレビュー工数を圧縮**。
これは「彼らの土俵で同じ人を増やす」のではなく「同じ品質を少ない人で出す」差別化。

## 4. $1M/month に向けた現実認識（正直版）

ツール単体では到達しない。必要なのは：
- **独占データ**（観点・欠陥DBの蓄積＝堀）← フェーズ2が起点
- **エンプラ連携と監査証跡**（第三者検証として通用する品質）← フェーズ1で土台、フェーズ3で実装
- **GTM/信頼**（実績・契約・ブランド）← 非エンジニアリング領域

本PRはこのうち**「データ資産化」と「監査証跡」の土台**を、AIなし・確立技術の範囲で先行実装したもの。
コードはビジネス全体の一部だが、“防御可能資産の起点”を最短で押さえた点に意味がある。

## 5. 推奨する次の一手

ここで「汎用ツールより当たる」ことを実データで示せれば、課金根拠と差別化が同時に立つ。
