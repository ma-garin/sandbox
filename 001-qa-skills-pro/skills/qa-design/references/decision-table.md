# デシジョンテーブルテスト（Decision Table Testing）

ISO/IEC/IEEE 29119-4 / ISTQB FL準拠。複数の条件の組み合わせによって動作が決まる
ビジネスルールを表に落とし、規則（列）ごとにテストケースを導出する。

## 適用手順

1. **条件と動作の抽出** — テストベースから条件（真偽 or 有限の値をとる判定）と動作（結果・出力・状態変化）を列挙する。各条件・各動作に仕様の根拠を付ける。
2. **完全表の構築** — 条件の全組み合わせ（真偽条件n個なら2^n列）をまず機械的に作る。**いきなり省略形から書かない**（組み合わせ漏れの温床）。
3. **簡約** — 結果に影響しない条件を「−」（don't care）でまとめ、実行不可能な組み合わせ（論理矛盾）は「実行不能」と根拠付きでマークして除外する。簡約前後の列数を宣言に記録する。
4. **矛盾・欠落の検出** — 以下を検出したら、テストケースより先に指摘として報告する（qa-review連携ポイント）:
   - 同じ条件組み合わせに異なる動作が定義されている（仕様の矛盾）
   - ある組み合わせに対する動作が仕様のどこにも書かれていない（仕様の欠落）
5. **テストケース化** — 実行可能な規則1列につき最低1ケース。条件値の具体化には同値分割の代表値を使う。

## テーブルの書式

```
条件                          | R1  | R2  | R3  | R4
------------------------------|-----|-----|-----|-----
C1: 会員である                | Y   | Y   | N   | N
C2: 購入額が5000円以上        | Y   | N   | Y   | N
------------------------------|-----|-----|-----|-----
A1: 10%割引                   | X   | X   |     |
A2: 送料無料                  | X   |     | X   |
```

- 条件・動作の各行に根拠（仕様該当箇所）を脚注で付ける。
- R列ごとに「根拠となる仕様の記述」を対応付ける。対応付けられない列は仕様欠落の指摘対象。

## カバレッジ計算（宣言に使用）

```
デシジョンテーブルカバレッジ = テストケースが実行する規則数 ÷ 実行可能な全規則数 × 100%
```

## よくある我流化（禁止パターン）

- 完全表を経由せず「ありそうな組み合わせ」だけ列挙する
- don't care の乱用で実際には結果が変わる組み合わせを潰す
- 仕様に動作が定義されていない列を、推測した動作で埋めて指摘せずに済ませる
