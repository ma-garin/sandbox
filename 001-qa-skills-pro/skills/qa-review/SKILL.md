# /qa-review — 仕様レビュー

仕様文書を入力とし、テスト可能性・一貫性・完全性の観点で欠陥を検出する。
AIが実装の真実の源泉として仕様を読む時代、仕様側の欠陥は実装欠陥として増幅される。
その増幅前に止めるのがこのSkillの役割。

## 手順

### 1. 共通ポリシーの読み込み

`../qa-shared/evidence-first-policy.md` と `../qa-shared/report-format.md` を読む。

### 2. 対象の構造把握

文書全体を読み、要求項目を列挙する。識別子（REQ-ID等）がない場合、
「要求に識別子がない」こと自体を最初の指摘とする（トレーサビリティ確保の前提欠落、severity: Medium以上）。

### 3. 観点別レビュー

以下の2参照ファイルを読み込み、観点ごとに全要求を走査する:

- `references/ambiguity-patterns.md` — 曖昧表現・未定義用語・測定不能表現の検出パターン
- `references/spec-review-checklist.md` — 完全性・一貫性・テスト可能性のチェックリスト

### 4. 指摘の検証

各指摘について出力前に自問する:
- 原文を引用できているか（引用できない指摘は破棄）
- 「文書内の別箇所で定義済み」ではないか（文書全体を確認してから指摘する）
- 修正案は仕様の意図を推測で上書きしていないか（意図が不明な場合、修正案ではなく「確認すべき質問」を出す）

### 5. 出力

`../qa-shared/report-format.md` の指摘構造で出力する。レポート構成:

1. サマリ（対象文書・版・指摘件数のseverity別内訳・実装着手可否の所見）
2. 指摘一覧（severity降順）
3. 確認すべき質問リスト（仕様作成者への質問。修正案が推測になる項目はこちらへ）
4. レビュー範囲の宣言（適用した観点・対象外にした観点を明記）

## 実装着手可否の所見基準

- **着手可**: Critical/High の指摘なし
- **条件付き着手可**: High はあるが該当機能を実装範囲から外せる
- **着手不可**: 中核機能の要求に Critical/High（曖昧・矛盾・欠落）が残っている

この所見は指摘の集計から機械的に導く。空気を読んで甘くしない。
