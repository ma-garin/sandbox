AIDD（AI-Driven Development）は、AIをペアプログラマとして活用しながら、品質・スピード・再現性を同時に高める開発スタイルです。このガイドでは3つのフェーズに分けて、ステップバイステップで学びます。

基本概念

AIDDとは何か

AI-Driven Development。AIに指示を出して成果物を作るのではなく、AIと対話しながら「仕様→設計→実装→テスト→振り返り」のループを高速に回す開発手法です。

従来の開発AIDD
「これを作って」と指示するspec.mdで要件を明示し、AIと対話しながら実装する
「動いた」で完了とするdone-gateチェックリストで「完了」を機械的に判定
レビューは感覚や経験に依存ISO 25010 + ISTQB severity で定量評価
ノウハウは個人の記憶に残るlessons.md + CURRENT_STATE.md で蓄積・引き継ぎ
毎回ゼロから始めるスキル・テンプレート・コマンドで再現性を確保
重要な概念： AIDDはAIへの「お願い」ではなく、AIとの「協業設計」です。仕様・品質基準・完了定義を明示することで、AIの出力を制御し、手戻りを防ぎます。

▼ design-system 初級
概要
手順
コード

何をするスキルか： AIDDツール群全体に一貫したビジュアルデザインを適用する設計システムです。色・フォント・スペーシング・コンポーネントのCSS変数が定義されており、全プロジェクトでこの変数を使うことで見た目の統一感を保ちます。

いつ使うか： UIを作るとき、色を決めるとき、新しいHTMLファイルを作り始めるとき。常に最初に参照します。

要素内容
ベースMaterial Design 3 / ライトテーマ
フォントNoto Sans JP（日本語）+ JetBrains Mono（コード）
管理CSS変数（:root）で一元管理
カラー数Primary / Surface / Border / Text / Status の5系統
  1. 1
    Google Fonts を head に追加する
    Noto Sans JP と JetBrains Mono を読み込みます。これだけで日本語が美しく表示されます。
  2. 2
    CSS変数（:root）をコピーする
    カラーパレット・タイポグラフィ・スペーシング・シャドウの変数をまるごとコピーし、スタイルの冒頭に貼ります。
  3. 3
    変数名をそのまま使う（上書き禁止）
    --color-primary を勝手に別の色に変えない。既存ツールとの統一感が崩れます。カスタマイズする場合は新しい変数名を追加します。
  4. 4
    レイアウトパターンを選ぶ
    「2ペイン（サイドナビ+メイン）」または「カードグリッド」のどちらかが既定パターン。そのままコピーして使います。
最初は変数を見て色を選ぶのでなく、既存ツール（QA Lens / UX Reviewer）のHTMLをそのまま参考にするのが早いです。

▼ nfr-standards 初級
概要
手順
テンプレート

何をするスキルか： 非機能要件（NFR）をspec.mdに書く際のテンプレートを提供します。パフォーマンス・デバイス対応・オフライン・セキュリティ・アクセシビリティの各要件が、プロジェクト種別ごとに整理されています。

いつ使うか： 新しいプロジェクトの spec.md を書くとき。「Lighthouseのスコアはどこを目標にする？」「Fold5対応が必要？」などを確認するとき。

種別条件
PWAGitHub Pages + Service Worker + localStorage
単一HTMLツール社内配布・ビルド不要・ブラウザで開くだけ
Streamlitアプリローカル/サーバ実行・Python
NFRを決めずに実装を始めると、後から「Fold5で崩れる」「オフラインで動かない」などの手戻りが発生します。spec.md作成時に必ず先に定義しましょう。
  1. 1
    プロジェクト種別を確認する
    PWA / 単一HTMLツール / Streamlit の3択。迷ったら「GitHub Pagesに公開するか」「スマホで使うか」で判断します。
  2. 2
    spec.mdの非機能要件セクションを作る
    種別に対応するNFRテンプレートをそのままコピーします。AIに「nfr-standardsスキルに従ってspec.mdの非機能要件を書いて」と依頼するだけでOKです。
  3. 3
    Fold5対応が必要か確認する
    PWAの場合はカバー画面（360×820px）での動作確認が必須。タッチターゲットは44px以上。
  4. 4
    APIキー管理方針を決める
    全種別でAPIキーのハードコード禁止。localStorage + 設定モーダルパターンを使います。

spec.mdの非機能要件セクション（PWA用最小版）

## 非機能要件

### 性能
- 初回ロード: 3G相当（1.6Mbps）で3秒以内に操作可能
- Lighthouse Performance: 80点以上

### デバイス対応
- Galaxy Z Fold5 カバー画面（360px）で主要機能が動くこと
- タッチターゲット: 44×44px以上

### オフライン
- Service Worker: cache-firstでコアUIをキャッシュ
- データはlocalStorageに保存。ネットワーク不要で全機能動作

### セキュリティ
- APIキーのハードコード禁止（localStorage保存）
- 外部依存はCDNのみ

### データ保全
- export（JSON）/ import機能を必ず実装
▼ single-html-tool 初級
概要
手順
パターン

何をするスキルか： 社内配布・デモ・QAツールを「単一のHTMLファイル」として作るための規約です。npm・ビルド不要でファイル1つをメールやSlackで送れば誰でも使えます。QA Lens・UX Reviewer はこのパターンで作られています。

いつ使うか： 「ブラウザで開くだけで動くツールを作りたい」「社内に配布したい」「デモを作りたい」とき。

絶対制約： CSS/JSの外部ファイル分割禁止。<style> と <script> をHTMLにインライン記述します。外部依存はCDNのみ許可。
機能実装パターン
データ永続化localStorage（キー: appName:dataType）
APIキー管理設定モーダル → localStorage保存
AI連携JSON強制出力、コードフェンス除去フォールバック
エクスポートCSV / Markdown / JSON / PDF印刷
  1. 1
    AIに依頼するときに「single-html-toolスキルに従って」と明示する
    「single-html-toolスキルに従って、〇〇ツールを単一HTMLで作って」と伝えるだけで、規約に沿った構造を生成してくれます。
  2. 2
    デザインシステムのCSS変数を冒頭に貼る
    <style>の先頭に :root { ... } の変数ブロックを置きます。
  3. 3
    localStorage操作を実装する
    保存: localStorage.setItem('appName:key', JSON.stringify(data))、読み込み: JSON.parse(localStorage.getItem('appName:key') || '[]')
  4. 4
    APIキーは設定モーダルで入力させる
    ハードコード厳禁。設定ボタン → モーダル表示 → 入力値をlocalStorageへ保存 → 以降は読み出して使用。
  5. 5
    ブラウザでダブルクリックして動作確認する
    done-gateの「単一HTMLツール追加項目」を確認します（この手順自体がdone-gateチェックの一部です）。

localStorage + APIキー管理の基本パターン

// APIキー取得
function getApiKey() {
  return localStorage.getItem('myTool:apiKey') || '';
}

// APIキー保存（設定モーダルのsaveボタンから）
function saveApiKey(key) {
  localStorage.setItem('myTool:apiKey', key);
}

// データ保存
function saveData(data) {
  const store = {
    version: 1,
    items: data,
    updatedAt: new Date().toISOString()
  };
  localStorage.setItem('myTool:data', JSON.stringify(store));
}

// データ読み込み
function loadData() {
  const raw = localStorage.getItem('myTool:data');
  if (!raw) return [];
  const store = JSON.parse(raw);
  return store.items || [];
}

▼ done-gate 初級
概要
チェックリスト

何をするスキルか： 「完了」と判定する前の最終チェックリストです。スクラムの Definition of Done に相当します。「動いた」と「完了」は別物。このゲートを通過して初めて「完了」です。

いつ使うか： 機能ができたと思ったとき。PRを出す前。タスクをクローズする前。

よくある間違い： 「動いた」「確認した」という主観的な判断で完了とすること。done-gateは「テストがpassする」「spec.md の検証基準を満たすevidenceがある」など客観的な事実で判定します。

全種別共通（毎回必ず確認）

  • ☐ spec.md の検証基準を全て満たす（evidence提示可能）
  • ☐ QAレビュー済み（Critical / High の指摘が残ゼロ）
  • ☐ テストがpassする（人力確認でなくテスト実行で判定）
  • ☐ 秘密情報がコードに混入していない（grepで確認）
  • ☐ CURRENT_STATE.md を更新した
  • ☐ コミット済み（Conventional Commits形式）

単一HTMLツールの追加項目

  • ☐ ブラウザでダブルクリックして動作確認した
  • ☐ CSS/JS外部分割していない（単一ファイル維持）
  • ☐ APIキーがハードコードされていない

PWAの追加項目

  • ☐ Fold5カバー画面（360px）で主要機能が動く
  • ☐ localStorage の export / import が動作する
  • ☐ Service Workerのキャッシュバージョンを更新した

使い方

  1. 1
    /doc-search コマンドを使う

▼ qa-review-standards 中級
概要
評価基準

何をするスキルか： ISO/IEC 25010（製品品質モデル）とISTQB severity分類を使って成果物をレビューします。「なんとなく良さそう」という主観を排除し、国際規格ベースで定量評価します。

いつ使うか： PRを出す前。done-gateのQAレビュー項目を実行するとき。/qa-reviewコマンドを呼んだとき。

ISO 25010の5カテゴリ評価する観点
機能適合性要件を満たしているか
使用性操作しやすいか、わかりやすいか
性能効率性速いか、リソースを無駄遣いしていないか
信頼性エラー処理が適切か、データが消えないか
セキュリティAPIキーが安全か、XSS等の脆弱性がないか
ISTQB Severity意味対応
Critical使用不能・データ消失・セキュリティホール即時修正必須。done-gate通過不可
High主要機能が動かない・大きな使いにくさ修正必須。done-gate通過不可
Medium機能は動くが品質に問題あり次のイテレーションで修正
Low軽微な改善点・好み余裕があれば修正
evidence-only原則： 根拠のない指摘はしない。「〇〇ページで△△ボタンをクリックしたとき、✕✕が発生した」という形でevidenceを提示します。
  1. 1
    /qa-review コマンドを呼ぶ
    対象の成果物（スクリーンショット・コード・仕様）を渡して「/qa-review」を実行。AIが5カテゴリ×severityでレビュー結果を出力します。
  2. 2
    Critical / High を先に修正する
    Medium / Low はあとでいい。まずCritical・Highをゼロにすることがdone-gateの条件です。
  3. 3
    delta比較でレビューする
    「前のバージョンと比べて何が変わったか」を明示することで、レビューの範囲を絞り込みます。
▼ test-automation 中級
概要
手順
コード例

何をするスキルか： 「動作確認した」を人力から自動テストの実行結果に置き換えるスキルです。PWA/HTMLツールはPlaywright、Streamlit/PythonはpytestでAIDDの品質ゲートに接続します。

いつ使うか： done-gateの「テストがpassする」を達成するとき。「毎回手動確認が面倒」と感じたとき。CIに組み込みたいとき。

種別フレームワーク最小テスト観点
PWA / 単一HTMLPlaywrightロード・主要操作・localStorage永続化・360px表示
Streamlitpytest + AppTest例外なく起動・session_state衝突なし・テナント越境なし
純Pythonロジックpytest関数の入出力・エッジケース
  1. 1
    AIに「test-automationスキルに従ってスモークテストを書いて」と依頼する
    対象ファイル（index.html や app.py）を渡すだけで、最小スモーク観点のテストコードを生成してくれます。
  2. 2
    tests/ フォルダにテストを配置する
    PWA: tests/test_smoke.py、Streamlit: tests/test_app.py
  3. 3
    ローカルで実行する
    pytest tests/ -v（Python/Streamlit）または pytest tests/test_smoke.py（Playwright）
  4. 4
    LLMを含むロジックはモックにする
    AIの応答はテストで再現できない（非決定的）ため、LLM呼び出しは必ずモック化してロジックだけをテストします。
time.sleep() でのテスト待機は禁止。Playwrightの expect / wait_for を使ってください。

▼ sdd-ecc-workflow 中級
概要
10ステップ
ガバナンス選択

何をするスキルか： 仕様駆動開発（Spec-Driven Development）でプロジェクトを進めるフローです。spec.mdを「単一の真実」として、実装と検証を分離して進めます。手戻りを90%削減する最重要スキルのひとつです。

いつ使うか： 「新しいプロジェクトを始めたい」「要件定義から進めたい」「タスク分解をしたい」とき。

役割何をするか
Coordinatorタスク割当・進捗管理のみ。コードを書かない
Implementortasks.mdの1タスクのみ実装。範囲外に手を出さない
Verifierspec.mdの検証基準に対して独立にevidenceで判定
1タスク1セッションが基本。タスクが終わったら /clear でセッションをリセットし、次のタスクから始めます。これがトークン節約の鍵です。
  1. 1
    spec.md を作成する
    要求・制約・検証基準を書く。「/sdd-start "概要一文"」で生成できます。
  2. 2
    plan.md を作成する
    アーキテクチャ・技術選定・ファイル構成を決める。
  3. 3
    tasks.md を作成する
    依存関係付きで実装タスクを分解。各タスクには「対象ファイル・検証基準・完了条件」を明記。
  4. 4
    実装ループを回す
    tasks.mdの1タスクを実装 → Verifierが検証 → 合格したら次のタスクへ。
  5. 5
    implement.md に記録する
    実装の判断・変更内容・ハマったことを残す。次セッションの文脈引き継ぎに使います。
  6. 10
    done-gate → retro で完了する
    チェックリストで完了判定 → lessons.md に振り返りを記録。

Phase 3 — AI・評価スキル 上級

目標：LLMベースのシステムを品質管理下で開発・運用する

▼ streamlit-rag-app 上級
概要
設計原則

何をするスキルか： PMOエージェント等の Streamlit RAGアプリ開発規約です。16モジュール構成・マルチテナント・3機能（意思決定支援・成果物支援・教育支援）の大規模アプリを品質基準を守りながら開発する方法を定義しています。

いつ使うか： Streamlit + LLMで業務アプリを作るとき。RAG（検索拡張生成）システムを構築するとき。

Phase 1・2のスキルが習得できてから取り組みましょう。特にsdd-ecc-workflow（仕様駆動）とtest-automation（テスト自動化）が前提です。
ポイント規約
モジュール構成1モジュール = 1責務。16モジュール以上に分割しない
session_stateキーは モジュール名_変数名 形式で衝突防止
LLM呼び出し1箇所のラッパー関数に集約（モデル切替・リトライを一元管理）
マルチテナントテナントIDを全DBクエリの第一キーとして強制フィルタ
  1. 1
    sdd-ecc-workflowでspec.mdを先に書く
    3機能（意思決定支援・成果物支援・教育支援）のうちどれを作るかを決め、要件を先に固めます。
  2. 2
    16モジュール構成を設計する
    app.py（起動）/ ui/（各ページ）/ core/（ビジネスロジック）/ data/（DB・RAG）など責務で分ける。
  3. 3
    LLMラッパーを最初に実装する
    core/llm.py に全LLM呼び出しを集約。モデル名・タイムアウト・リトライをここで管理。
  4. 4
    マルチテナントテストを書く
    テナントAのデータがテナントBから見えないことをpytestで検証。これが必須のdone-gate項目です。
▼ agent-eval 上級
概要
評価フロー

何をするスキルか： AIエージェント・LLMシステムの出力品質を定量評価します。DeepEval（pytest-native）+ Langfuse（トレース・UI）を使って、非決定的なAI出力を測定可能にします。

いつ使うか： プロンプト・モデル・ロジックを変更したとき。AI出力の品質を数値で把握したいとき。回帰テストを設定したいとき。

ツール役割
DeepEval評価メトリクス実装（Faithfulness / Relevancy / Correctness 等）
LangfuseLLM呼び出しのトレース・コスト・レイテンシの可視化
ゴールデンデータセット「正解がわかっている入出力ペア」の集合。評価の基準
test-automationとの違い： test-automationは「コードが動くか（決定的）」を確認します。agent-evalは「AIの回答が良いか（非決定的）」を定量評価します。両方必要です。
  1. 1
    ゴールデンデータセットを作る
    「この入力に対してこの回答が正解」というペアを最低10件用意します。ここが最も重要で、品質の基準になります。
  2. 2
    評価メトリクスを選ぶ
    RAGなら Faithfulness（根拠に忠実か）+ Relevancy（関連性あるか）が最低限。タスクに応じて Correctness・Coherence を追加。
  3. 3
    ベースラインスコアを記録する
    最初の evalを「ベースライン」として保存。以降のプロンプト変更で回帰していないかこの値と比較します。
  4. 4
    /eval コマンドで定期実行する
    プロンプト・モデル変更のたびに /eval を実行し、スコアがベースライン以上であることを確認してから done-gate を通過させます。

ファイル役割更新タイミング
CURRENT_STATE.mdセッション間の文脈引き継ぎ（最重要）毎セッション終了時（5分）

CURRENT_STATE.md テンプレート

Conventional Commits 形式

1コミット = 1論理変更 を守りましょう。複数の変更をまとめて1コミットにしない。小さくコミットするほど、問題が起きたときの切り分けが簡単になります。
