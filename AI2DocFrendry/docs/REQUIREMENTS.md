## 1. システム概要

多様な拡張子で散在する社内ナレッジを投入すると、**生成AI（RAG）が読み込みやすい形式へ変換**し、**変換結果のAI可読性を採点**して**改善提案を返す**、ローカル完結型のRAG前処理支援システム。

```
[社内ナレッジ]              [AI2DocFrendry]                    [出力]
 docx/xlsx/pptx  ──┐                                    ┌── Markdown（構造化済み）
 pdf/txt/md/csv  ──┼─→ 取込 → 抽出 → 正規化 → 構造化 ─┼── JSONL（チャンク済み・RAG投入用）
 html/eml        ──┘         ↓                          ├── metadata.csv（一覧・出典・機密区分）
                          採点・診断  ────────────────→ └── 品質レポート（スコア＋指摘＋改善提案）
```

## 2. 利用者（ペルソナ）

| ID | 利用者 | 立場 | 主な関心 |
|---|---|---|---|
| U-1 | ナレッジ管理者 | 情シス／QA-PMO | 大量ファイルを一括で変換したい。何が変換できて何ができなかったかを知りたい |
| U-2 | RAG構築担当 | 開発者 | そのまま投入できるチャンク済みデータが欲しい。品質が担保されているか確認したい |
| U-3 | 文書作成者 | 現場担当者 | 自分の文書がAIに読まれるのか、どう直せばよいかを知りたい |

## 3. 業務要求（Business Requirements）

| ID | 要求 | 対応する課題 |
|---|---|---|
| BR-01 | 多形式のナレッジを、形式を意識せず一括で投入できること | P-01 |
| BR-02 | 変換後の文書が、元の構造（見出し・表・箇条書き）を保っていること | P-02 |
| BR-03 | RAGに投入する**前に**、AIが読める状態かを判定できること | P-01〜P-04 |
| BR-04 | 読めない・読みにくい文書を特定し、作成者へ改善を促せること | P-03 |
| BR-05 | 機密情報・陳腐化・重複を投入前に検知できること | P-05, P-06 |
| BR-06 | 社内ナレッジを外部へ送信しないこと | 情報管理規程 |
| BR-07 | 追加コスト0円で運用できること | 予算制約 |

## 4. 機能要件（Functional Requirements）

> 優先度: **Must**（MVP必須）／**Should**（MVP後の早期）／**Could**（将来）

### 4-1. 取り込み

| ID | 要件 | 優先度 |
|---|---|---|
| FR-01 | フォルダ（再帰探索）、ZIP、個別ファイルのいずれでも投入できること | Must |
| FR-02 | 対象拡張子・除外パターン（例: `~$*`, `.git/`）を設定できること | Must |
| FR-03 | 対応外形式・破損ファイルは処理を止めず隔離し、理由付きで一覧化すること | Must |
| FR-04 | 同一ファイルの再投入時、内容ハッシュが同一なら再処理をスキップできること | Should |

### 4-2. 抽出

| ID | 要件 | 優先度 |
|---|---|---|
| FR-05 | 次の形式からテキストと構造を抽出できること：`.docx` `.xlsx` `.pptx` `.pdf` `.txt` `.md` `.csv` `.html` | Must |
| FR-06 | 見出し階層、箇条書き、表、脚注を構造情報として保持すること | Must |
| FR-07 | 表はセル結合を展開し、行・列の対応が崩れない形で取り出すこと | Must |
| FR-08 | 画像・図形は本文から除外し、「画像あり（本文非抽出）」として位置と件数を記録すること | Must |
| FR-09 | `.eml` `.msg`（メール）から件名・日時・本文・添付一覧を抽出できること | Should |
| FR-10 | 画像内文字・スキャンPDFをOCRで抽出できること | Could |
| FR-11 | 旧形式（`.doc` `.xls` `.ppt`）を変換して取り込めること | Could |

### 4-3. 正規化・構造化

| ID | 要件 | 優先度 |
|---|---|---|
| FR-12 | 文字コードを判定しUTF-8へ統一すること（判定不能時は隔離） | Must |
| FR-13 | ヘッダ・フッタ・ページ番号・目次など、全ページ反復するノイズを除去すること | Must |
| FR-14 | 抽出した構造をMarkdownへ再構成すること（見出し`#`、表はMarkdown表、箇条書き`-`） | Must |
| FR-15 | 各文書にYAML front matterでメタデータを付与すること（出典パス、拡張子、タイトル、更新日、ハッシュ、機密区分、可読性スコア） | Must |
| FR-16 | 表記ゆれ（全半角、長音、送り仮名）を検出すること。**自動修正はせず検出のみ**（ADR-04） | Must |

### 4-4. チャンク分割

| ID | 要件 | 優先度 |
|---|---|---|
| FR-17 | 見出し境界を優先して分割し、上限トークン数を超える場合のみ二次分割すること | Must |
| FR-18 | チャンクに `chunk_id` / `parent_doc_id` / 見出しパス / 出典を付与すること | Must |
| FR-19 | チャンクサイズ・オーバーラップ量を設定できること | Must |
| FR-20 | 表は可能な限り分断せず、1チャンクに収めること（超過時は見出し行を各チャンクへ複製） | Should |

### 4-5. AI可読性スコアリング（本システムの中核）

| ID | 要件 | 優先度 |
|---|---|---|
| FR-21 | 文書ごとに6観点（構造明示性／自己完結性／表現一貫性／抽出性／文脈情報量／安全性）を0〜100で採点すること | Must |
| FR-22 | 総合スコアを算出し、A〜Dの判定（そのまま投入可／要確認／要改善／投入非推奨）を返すこと | Must |
| FR-23 | 指摘はISTQB severity（Critical/High/Medium/Low）で分類すること | Must |
| FR-24 | 全指摘に根拠（該当箇所の引用と位置）を添えること。**根拠のない指摘は出さない（evidence-only）** | Must |
| FR-25 | 指摘ごとに具体的な改善アクションを提示すること（例:「『別紙参照』を実体の記述に置き換える」） | Must |
| FR-26 | 採点は決定的であること（同一入力に対し常に同一出力） | Must |
| FR-27 | 観点別の重み・しきい値を設定で変更できること | Should |
| FR-28 | LLMによる補足診断を任意で有効化できること（既定OFF、キーはlocalStorage相当の外部保持） | Could |

### 4-6. 安全性・鮮度チェック

| ID | 要件 | 優先度 |
|---|---|---|
| FR-29 | 機密情報らしきパターン（メールアドレス、電話番号、マイナンバー形式、APIキー、パスワード表記）を検出し警告すること | Must |
| FR-30 | 検出パターンを設定ファイルで追加・除外できること | Must |
| FR-31 | 最終更新日から一定期間（既定3年）を超える文書を陳腐化候補として警告すること | Must |
| FR-32 | 内容ハッシュ一致による完全重複、および高類似度の文書を検出すること | Should |
| FR-33 | 検出した機密情報をマスクした出力を生成できること | Could |

### 4-7. 出力

| ID | 要件 | 優先度 |
|---|---|---|
| FR-34 | 文書ごとのMarkdownを出力すること | Must |
| FR-35 | チャンク済みJSONL（1行1チャンク、メタデータ付き）を出力すること | Must |
| FR-36 | 一覧CSV（出典、形式、スコア、判定、指摘件数、機密検知、陳腐化）を出力すること | Must |
| FR-37 | 品質レポート（全体サマリ＋文書別スコア＋指摘一覧）を出力すること | Must |
| FR-38 | 上記一式をZIPで一括ダウンロードできること | Must |
| FR-39 | 変換前後をUI上で並べて確認できること | Should |

### 4-8. 実行・運用

| ID | 要件 | 優先度 |
|---|---|---|
| FR-40 | 処理の進捗（件数・残数・現在処理中のファイル）を表示すること | Must |
| FR-41 | 処理ログ（ファイル単位の成否と理由）を残すこと。**本文はログに残さない** | Must |
| FR-42 | 中断後、未処理分から再開できること | Should |
| FR-43 | CLIから実行できること（定期実行・CI連携用） | Should |

## 5. 非機能要件（ISO/IEC 25010）

| ID | 品質特性 | 要件 | 測定・受入基準 |
|---|---|---|---|
| NFR-01 | 機能適合性 | 主要8形式で本文と表を欠落なく抽出できること | 検証用サンプル各形式3件で抽出率95%以上 |
| NFR-02 | 性能効率性 | 1,000ファイル（平均500KB）を30分以内に処理すること | 実測。1ファイル平均3秒以内 |
| NFR-03 | 性能効率性 | メモリ使用量が2GBを超えないこと（逐次処理・全件メモリ展開の禁止） | 実測 |
| NFR-04 | 互換性 | macOS（M1）とWindows 11 の双方で動作すること | 両環境でスモークテスト通過 |
| NFR-05 | 使用性 | 投入から出力まで3ステップ以内で完了すること | UI操作手順の実測 |
| NFR-06 | 使用性 | 専門知識なしに判定結果（A〜D）と改善アクションを理解できること | U-3による受入確認 |
| NFR-07 | 信頼性 | 1ファイルの失敗が全体処理を停止させないこと | 破損ファイル混入テストで継続を確認 |
| NFR-08 | 信頼性 | 同一入力に対し常に同一出力となること（冪等・決定的） | 同一入力2回実行の出力ハッシュ一致 |
| NFR-09 | **セキュリティ** | 既定構成で外部ネットワーク通信を一切行わないこと | 通信監視で送信ゼロを確認（**最重要**） |
| NFR-10 | セキュリティ | ログ・エラーメッセージに文書本文および機密情報を出力しないこと | ログ検査 |
| NFR-11 | セキュリティ | APIキー等の認証情報をソースへ埋め込まないこと | 静的検査・コードレビュー |
| NFR-12 | 保守性 | パーサをプラグイン構造とし、形式追加を既存改変なしで行えること | 新規形式追加時の差分が新規ファイルのみ |
| NFR-13 | 保守性 | 1ファイル400行以内、関数50行以内。処理系は純粋関数とすること | レビュー・静的検査 |
| NFR-14 | 保守性 | 単体テストカバレッジ80%以上（スコアリング処理は必須） | 計測 |
| NFR-15 | 移植性 | `venv` と `requirements.txt` のみで環境構築が完了すること | クリーン環境で構築確認 |

## 7. 前提条件

| ID | 前提 |
|---|---|
| ASM-01 | 投入対象は利用者が閲覧権限を持つファイルであり、権限管理はシステムの責務外 |
| ASM-02 | 機密情報検出は補助機能であり、最終的な公開可否判断は利用者が行う |
| ASM-03 | 技術スタックは Python 3.11+ / Streamlit を想定（[`RFD.md`](RFD.md) 選択肢B） |
| ASM-04 | MVPではOCRを行わないため、スキャンPDFは「抽出性スコア低」として検出されるに留まる |

## 8. 受入基準（発注者視点）

| ID | 基準 |
|---|---|
| AC-01 | 実在の社内ナレッジ100件を投入し、すべてのファイルが「変換完了」または「理由付きで隔離」のいずれかに分類されること |
| AC-02 | 変換後Markdownの見出し・表が、原本と目視で一致すること（サンプル20件） |
| AC-03 | 意図的に低品質な文書（ベタ書き・別紙参照・スキャンPDF）が、いずれもC判定以下となること |
| AC-04 | 機密情報を埋め込んだテストファイルがCritical指摘として検出されること |
| AC-05 | 実行中に外部通信が発生しないこと |
| AC-06 | 出力JSONLがそのままRAG投入可能な形式であること（スキーマ検証を通過） |
