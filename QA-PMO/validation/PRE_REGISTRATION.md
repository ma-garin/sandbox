実際に本番で発生したドメイン固有バグをより多く事前に捕捉できる」

## 3. バグ選定基準（事前固定）

- ラベル: `type: Bug` または `Priority: P1/P2/Critical` または `type: task` with bug description
- 状態: Closed（修正確認済み = 本当に起きたバグ）
- 期間: 直近2年分（2023年〜2024年）
- 対象カテゴリ: Checkout / Payment / Order / Inventory / Discount / Tax
- 件数: 上記条件で上位30件を機械的に取得（件数で恣意的に絞らない）
- 除外: ドキュメント/CI/インフラのみのバグ（コード動作に影響しないもの）

## 5. 採点ルール（事前固定）

### 「捕捉」の定義
テスト条件Xが、実バグBを「捕捉した」とみなす基準：
> 「もしテスト条件Xを実行していたら、バグBが発現する状況を
> 意図的に再現するケースが含まれていた」と合理的に判断できる場合。

### 採点者・偏り除去
- 採点者1: Claude（主採点）
- 採点者2: ユーザー（サンプル20%を独立して採点）
- 不一致時: 保守的側（捕捉「なし」）を採用
- 疑わしい場合は「捕捉なし」で判定（自分に不利な方向で解決）

### 測定指標
- **全バグ捕捉率**（ただしこれは参考値）
- **ドメイン固有バグ捕捉率**（主指標 ← 我々の勝ち筋）
  - ドメイン固有 = Concurrency / State / Data / Security / I18N カテゴリ
