# 仕様レビューチェックリスト（完全性・一貫性・テスト可能性）

IEEE 29148（要求工学）の品質特性を実務向けに再構成したチェックリスト。
各項目は「Yes/No で判定でき、No なら原文（または不在確認）を根拠に指摘できる」ように書いてある。

## A. 完全性（Completeness）

- [ ] A1: 全機能要求に、正常系の入力・処理・出力が揃って書かれているか
- [ ] A2: 全機能要求に、異常系（入力不正・外部失敗・権限なし）の動作が定義されているか
- [ ] A3: 非機能要求（性能・信頼性・セキュリティ・アクセシビリティ）のセクションが存在するか。存在しない場合、それ自体を指摘（不在確認を根拠とする）
- [ ] A4: データのライフサイクル（作成・更新・削除・保持期間・削除時の連鎖）が定義されているか
- [ ] A5: 画面・APIの一覧と要求の対応が取れるか（要求に現れるのに一覧にない画面／その逆）

## B. 一貫性（Consistency）

- [ ] B1: 同一の値・制限（上限数・期限・金額）が文書内で一致しているか
- [ ] B2: 用語が用語集または初出定義と一貫して使われているか
- [ ] B3: 関連文書（design.md・API定義・画面設計）と矛盾しないか（参照可能な場合のみ。参照できない文書は「未確認」と宣言する）
- [ ] B4: 状態や権限のモデルが章をまたいで一貫しているか（3章では4状態、5章では5状態、等がないか）

## C. テスト可能性（Verifiability）

- [ ] C1: 各要求は、合否を判定できる検収条件に書き換え可能か（ambiguity-patterns.md の観点1・3で走査）
- [ ] C2: 期待結果が観測可能か（「内部的に最適化する」等、外から検証できない要求がないか）
- [ ] C3: 前提条件・テストデータが準備可能か（本番データ前提・実在人物のデータ前提になっていないか）

## D. トレーサビリティ（Traceability）

- [ ] D1: 要求に一意な識別子があるか（REQ-XX等）
- [ ] D2: 要求の由来（誰の・何のための要求か）が追えるか
- [ ] D3: 優先度 or 必須/任意の区別があるか（全部必須は区別がないのと同じ）

## E. AIDD固有の観点

- [ ] E1: AIが推測で埋めそうな穴がないか — TBD・「いい感じに」・暗黙の業界常識への依存。仕様が沈黙している箇所は、AIにとって「自由に決めてよい箇所」になる
- [ ] E2: 仕様の鮮度 — 最終更新日と、参照しているコード・外部仕様の版が明記されているか。古い仕様を根拠にAIが実装する事故の予防
- [ ] E3: 実装例・サンプルコードが仕様本文と矛盾していないか（AIはサンプルコードを仕様より優先して模倣しがち）

## 運用ルール

- チェックは全項目実施し、レポートの「レビュー範囲の宣言」に実施項目と未実施項目（理由付き）を列挙する。
- No の項目だけ指摘化する。Yes の項目は列挙しない（ノイズ削減）。
- 1指摘1項目。複数観点に該当する場合は最も重い観点で1件にまとめ、関連観点を本文に書く。
