# RFD — AI2DocFrendry（RAGサポートシステム）

> RFD = Request For Discussion。実装前に「何を作らないか」を含めて合意するための討議文書。
> ステータス: **ドラフト（レビュー待ち）** ／ 版: 0.2 ／ 起票日: 2026-07-21 ／ 起票者: ma-garin

---

## 1. 背景

社内ナレッジは、Word・Excel・PowerPoint・PDF・メール・テキスト・HTMLなど**多様な拡張子に分散**して蓄積されている。これらをそのまま生成AI（RAG）に投入すると、次の問題が起きる。

| # | 問題 | 具体例 | 原因の所在 |
|---|---|---|---|
| P-01 | 形式ごとに読み取り可否がばらつく | スキャンPDFは文字が取れず、実質「空のファイル」として投入される | 形式 |
| P-02 | 構造が失われる | Excelの結合セル、PowerPointのテキストボックスが意味のない文字列の羅列になる | 形式 |
| P-03 | **そもそも文書が構造化されていない** | 全文がベタ書きで見出しがなく、どこに何が書いてあるか機械的に判別できない | **中身** |
| P-04 | **文書が自己完結していない** | 「上記の通り」「標記の件」「別紙参照」がチャンク分割で文脈を失い、AIが誤答する | **中身** |
| P-05 | ゴミが混入する | ヘッダ/フッタ、ページ番号、目次が全チャンクに紛れ込みノイズになる | 形式 |
| P-06 | 表記がゆれる | 「ユーザ」「ユーザー」「user」が混在し、検索でヒットしない | 中身 |
| P-07 | 危険な情報が混入する | 個人情報・APIキー・機密区分の高い文書が無検査で投入される | 運用 |
| P-08 | 陳腐化・重複が判別できない | 旧版と新版が同時にヒットし、AIが古い情報を答える | 運用 |

**P-03・P-04・P-06 は形式変換では解決しない。**docxをMarkdownに変換しても、ベタ書きはベタ書きのままであり、「上記の通り」は「上記の通り」のまま残る。ここが本システムの主戦場である。

## 2. 課題設定

> 「多形式の社内ナレッジを、生成AIが読める形に**変換する**手段がない。
>  形式変換だけなら既存OSSで足りるが、**中身が構造化されていない文書はAIに読めないまま**である。」

## 3. 提案

**AI2DocFrendry** — 社内ナレッジのファイル群を投入すると、

1. **形式変換** — 多形式からMarkdownへ、構造（見出し・表・箇条書き）を保って変換する
2. **再構成** — 中身をAIが読める形に作り直す（見出しの付与、指示語・参照の実体化、表記ゆれの統一、メタ情報の補完）★中核
3. **改変検証** — 再構成が原文の情報を歪めていないかを機械的に検証し、違反した変更は自動で取り消す
4. **可読性診断** — 再構成後もなお残る問題を、根拠付きで報告する
5. **RAG投入形式で出力** — チャンク済みJSONL＋Markdown＋変更差分

ローカル完結型のRAG前処理支援システム。

### 3-1. 何をもって「AIが読める形」とするか

| # | 観点 | 変換前の状態 | 再構成後の状態 |
|---|---|---|---|
| V-1 | 構造明示性 | 全文が1段落のベタ書き | 見出し階層・箇条書き・表としてマークアップされている |
| V-2 | 自己完結性 | 「上記の通り」「標記の件」「別紙参照」 | チャンク単体で意味が通る記述に置き換わっている |
| V-3 | 表現一貫性 | 「ユーザ」「ユーザー」が混在 | 文書内で統一されている |
| V-4 | 抽出性 | 画像内の表、スキャンPDF | テキストとして取り出せている（不能なら明示的に警告） |
| V-5 | 文脈情報量 | 版数・日付・所管の記載がない | ファイル属性等から補完され、front matterに記録されている |
| V-6 | 安全性 | 機密情報・陳腐化・重複が未検知 | 検知され警告されている |

診断（4）はこの6観点を**変換前と再構成後の両方で採点**し、「どれだけ読める形になったか」を示す。診断は成果物ではなく、**再構成の効果と限界を伝えるための添え物**である。

### 3-2. スコープ外（作らないもの）

| # | 作らないもの | 理由 |
|---|---|---|
| X-01 | ベクトルDB・検索・回答生成 | LangChain/LlamaIndex等の既存領域。本システムは**その手前**に徹する |
| X-02 | 形式変換エンジンの自作 | 既存OSSで足りる。委譲する |
| X-03 | ナレッジ管理UI（保管・権限・ワークフロー） | SharePoint等の領域 |
| X-04 | **要約** | 情報が失われる。再構成は情報量を保ったまま構造を変えることに限る |
| X-05 | **原文にない情報の追加** | 事実の創作にあたる。検証（3-3）で機械的に禁止する |
| X-06 | 原本の上書き | 原本は常に保持し、成果物は別ファイルとして出す |

### 3-3. 再構成の3原則（本システムの安全装置）

再構成は「AIに社内文書を書き換えさせる」行為であり、放置すれば**事実の捏造・情報の欠落**を招く。以下を機械的に強制する。

| # | 原則 | 実装上の担保 |
|---|---|---|
| RP-1 | **情報を追加しない** | 再構成後に出現する固有名詞・数値・日付が原文に存在するか照合。不一致なら当該変更をロールバック |
| RP-2 | **情報を削除しない** | 原文の各文が再構成後に対応を持つかを照合。カバー率が閾値未満ならロールバック |
| RP-3 | **すべての変更を差分で提示する** | 変更種別・方式（ルール／LLM）・信頼度を付けて `diff.md` に出力。人がレビューできる状態にする |

> **ロールバックは安全側に倒す**。検証を通らなかった変更は適用せず、原文のまま残す。「直しすぎて間違える」より「直しきれずに残る」ほうを選ぶ。

## 4. 判断軸と選択肢比較

### 判断軸（優先順）
1. **機密安全性** — 社内ナレッジを外部に送信しないこと（最優先）
2. **原文忠実性** — 再構成が事実を歪めないこと（★v0.2で追加。再構成を行う以上、最重要級）
3. **コスト** — 0円（無料OSSのみ、外部登録なし）
4. **再現性** — 同じ入力から同じ出力が得られること
5. **導入容易性・拡張性**

### 選択肢A：形式変換の方式

| 案 | 内容 | 機密 | コスト | 判定 |
|---|---|---|---|---|
| A-1 | **個別パーサ（python-docx / openpyxl / python-pptx / pdfplumber）をプラグイン構造で** | ◎ | ◎ | **採用** |
| A-2 | MarkItDown / Docling 等の統合OSSに委譲 | ◎ | ◎ | フェーズ2で精度評価し差し替え可能にする |
| A-3 | クラウドAPI（LlamaParse / Azure DI） | × 外部送信 | × 従量課金 | 却下 |

### 選択肢B：**再構成の実現方式**（v0.2の中心論点）

見出しの生成や指示語の実体化は、**内容を理解しないと実行できない**。ルールベースには原理的な限界がある。

| 案 | 内容 | 機密 | 原文忠実性 | 再現性 | コスト | できること |
|---|---|---|---|---|---|---|
| B-1 | ルールベースのみ | ◎ | ◎ | ◎ 完全 | ◎ | 表記ゆれ統一、表整形、章番号参照の解決、メタ補完、略語定義。**見出し生成と指示語解決は不可** |
| B-2 | クラウドLLM（Claude/GPT API） | **× 外部送信** | △ | △ | × 従量 | すべて可能 |
| B-3 | **ローカルLLM（Ollama等）** | ◎ 送信なし | △→検証で担保 | ○ seed固定 | ◎ 無料 | すべて可能 |
| B-4 | **B-1 と B-3 の二層構造** | ◎ | ◎ 検証で担保 | ○ | ◎ | ルールで確実に直せるものはルールで、残りをLLMで |

→ **B-4を採用。**

```
層1（ルールベース・決定的・必須）
  表記ゆれ統一／表の整形／章番号参照の解決／メタ情報補完／略語の初出定義／ノイズ除去
        ↓ ここまでで直しきれないもの
層2（ローカルLLM・任意・既定ON／オフライン）
  見出しの生成／指示語の実体化／箇条書き化
        ↓ すべての変更に対して
層3（改変検証・ルールベース・決定的・必須）
  情報追加なし／情報欠落なし／数値同一性 → 違反した変更はロールバック
```

- 層2を無効化しても**システムは動く**（層1のみで完結する）。LLMが使えない環境でも導入できる。
- 層2の出力は必ず層3の検証を通る。**LLMに自由を与えず、検証で縛る。**
- 判断軸1（機密）はローカル実行で守り、判断軸2（忠実性）は層3で守る。

### 選択肢C：UI形態

| 案 | 導入容易性 | 適性 | 判定 |
|---|---|---|---|
| C-1 | **Streamlit（Python）** | ○ venv一発 | ◎ 大量ファイル処理・進捗表示・差分表示に適する | **採用** |
| C-2 | 単一HTMLツール | ◎ | × ローカル大量走査・Officeパース・LLM連携が不可能 | 却下 |
| C-3 | Django | △ | △ バッチ処理主体でDB不要 | 却下 |

→ **C-1。** `QA-knowledge` と同型でCLIも併設する。

## 5. 決定事項（ADR）

| ID | 決定 | 理由 |
|---|---|---|
| ADR-01 | 全処理をローカルで完結させ、外部送信を既定で行わない（LLMもローカル実行） | 社内ナレッジは機密を含む。判断軸の最上位 |
| ADR-02 | 中間形式・成果物はMarkdown（＋YAML front matter） | LLM入力の事実上の標準。人間もレビューできる |
| ADR-03 | チャンクは見出し境界を優先し、トークン上限で二次分割する | 意味のまとまりを保つため |
| ADR-04 | **本文を再構成する。ただし原本を保持し、全変更を差分提示し、原文にない情報の追加を機械検証で禁止する** | v0.1の「提案までに留める」を反転。ただし無条件の書き換えは認めない（3-3） |
| ADR-05 | 再構成は「層1ルールベース → 層2ローカルLLM → 層3検証」の三層とし、層2は無効化可能とする | LLMなしでも動作させ、LLMの出力は必ず検証で縛るため |
| ADR-06 | 要約と情報追加を行わない（構造のみを変える） | 情報量の保存。RAGの情報源として原文と等価であることを保証する |
| ADR-07 | 検証を通らない変更は適用せず原文のまま残す（安全側ロールバック） | 直しすぎによる誤りより、直しきれない残存を選ぶ |
| ADR-08 | パーサ・再構成操作はプラグイン構造とする | 形式・操作の追加を実装追加のみで可能にする |
| ADR-09 | 可読性スコアは変換前・再構成後の両方を算出する | 再構成の効果と限界を可視化するため |

## 6. リスクと対応

| # | リスク | 影響 | 対応 |
|---|---|---|---|
| RSK-01 | **LLMが事実を捏造する（ハルシネーション）** | 誤った社内情報がRAGに載る | **層3の改変検証で機械的に検出しロールバック**（RP-1）。最重要対策 |
| RSK-02 | 再構成で情報が欠落する | 원문にあった内容が失われる | 層3のカバー率検証（RP-2）。要約は行わない（ADR-06） |
| RSK-03 | ローカルLLMの品質・速度が実用に足りない | 層2が使えない | 層1のみで動作する設計とする。モデル選定はW-6で実測 |
| RSK-04 | スキャンPDF・画像内文字が取得できない | 中身が空のまま投入される | MVPではOCRせず「抽出性Critical」として検出・警告。OCRはフェーズ2 |
| RSK-05 | 旧形式（.doc/.xls/.ppt）が読めない | 対象漏れ | MVPは対象外。一覧に出す。LibreOffice CLI連携はフェーズ2 |
| RSK-06 | 大量ファイルで処理が終わらない（LLM併用で顕著） | 実用性を損なう | 逐次処理＋中断再開＋層2のON/OFF＋対象文書の絞り込み（低スコア文書のみLLM適用） |
| RSK-07 | 再構成の妥当性が検証されていない | 出力が信用されない | `QA-PMO/validation/` に倣い、実ファイルでの妥当性検証工程（W-16）を必須とする |
| RSK-08 | 機密情報検出の見逃し（偽陰性） | 情報漏洩 | 「検出は補助であり最終責任は利用者」と明示。パターンは設定で追加可能 |

## 7. 討議したい点（レビュー依頼事項）

| # | 論点 | 現時点の案 |
|---|---|---|
| D-01 | 層2のローカルLLMを既定ONとするか。導入時にモデルのダウンロードが必要になる | 既定ON。未導入時は自動的に層1のみで動作 |
| D-02 | 対応拡張子の優先順位。実際の社内ナレッジの分布は？ | docx / xlsx / pptx / pdf / txt / md / csv / html を第1優先 |
| D-03 | 再構成結果を人がレビューする運用を挟むか、自動で投入まで進めるか | 差分を出力し、レビューは利用者の運用に委ねる |
| D-04 | 利用者は誰か | ナレッジ管理者を主、RAG構築担当を従 |
| D-05 | プロジェクト名。`Frendry` は `Friendly` の意図か | 現状はフォルダ名のまま据え置き |
