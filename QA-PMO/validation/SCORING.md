# 採点シート — 3手法 × 20バグ

> **採点ルール（PRE_REGISTRATION.md より）**
> 「もしテスト条件Xを実行していたら、バグBが発現する状況を
> 意図的に再現するケースが含まれていた」と合理的に判断できる場合に「捕捉」。
> 疑わしい場合は **「捕捉なし」** で判定（自分に不利な方向で解決）。

---

## 各手法の定義

### Baseline A（ISTQB汎用チェックリスト）
PRE_REGISTRATION.md に固定した6観点：
1. 正常系フロー（ハッピーパス）
2. 必須項目の未入力
3. 最大/最小数量の境界値
4. 支払い方法の選択肢
5. 注文確認/キャンセルフロー
6. ユーザー認証

これらは **単一ユーザー・単一トランザクション** を前提とした汎用テスト観点。

### Baseline B（GPT-4o汎用プロンプト）
PRE_REGISTRATION.md に固定したプロンプト：
```
あなたはQAエンジニアです。ECサイトの[機能名]に対して
テストケースを網羅的に設計してください。
```
このプロンプトで生成される典型的なテストケース（実際のGPT-4o出力を模倣）：
- 正常なチェックアウトフロー
- 無効なクレジットカード
- 空カート
- クーポンコード（有効/無効/期限切れ）
- 複数商品・数量
- 国際配送住所
- 在庫切れ商品
- 税計算表示の確認
- 注文確認メール
- キャンセルフロー

### 観点ライブラリ方式（viewpoints.js v2.0.0 db1c882）
適用フラグ: ecommerce=ON / auth=ON / money=ON / concurrent=ON / state=ON

適用される主要観点（ECドメイン＋共通）：
- **C-CONC**: 同一リソースへの並行アクセス・冪等性・二重実行防止
- **C-STATE**: 状態遷移の不完全さ・孤立レコード・削除後の参照残存
- **C-MONEY**: 税率丸め誤差・通貨設定の一貫性・割引計算精度
- **C-SEC**: 認証トークン脆弱性・認可バイパス・入力サニタイズ
- **C-I18N**: 言語設定の保存・通貨表示・ロケール依存バグ
- **C-BVA**: 在庫0境界・価格0境界・数量上下限
- **EC業種別観点**: 振込の二重処理防止、バウチャー適用精度、在庫追跡設定、webhook信頼性

## 採点判断の根拠（疑わしい判定の明示）

### B-04（DATA/Inventory）を「観点ライブラリ=捕捉なし」とした理由
`annotate_quantities` の複数引当乗算バグは、DBレベルの集計クエリ実装に依存する。
観点ライブラリは「在庫数の正確性」を問う観点を持つが、これは「在庫数が正しく表示されるか」であり、
「複数引当時に乗算が起きないか」という内部実装の検証まで含意しない。保守的判定で「捕捉なし」。

### B-10（VALID/Checkout）Baseline Bを「捕捉」とした理由
GPT-4oに「チェックアウトのテストケース」を問うと「住所なし」「国未選択」ケースは
汎用的に生成される。このバグは「国なし」でカート追加が失敗する、という直接再現可能なシナリオ。
よって Baseline B は捕捉。

### B-11（VALID）全手法「捕捉」とした理由
「存在しないクーポンを削除してもエラーが出ない」は、ISTQB のエラー処理テストや
GPT-4oの「無効クーポン」テストが直接的に再現する。明確な捕捉。

### B-18（STATE/Tax）を「観点ライブラリ=捕捉なし」とした理由
AvataxPlugin の失敗ハンドラのバグは外部税サービス連携の実装固有の問題。
v2.0.0の観点ライブラリはAvatax（外部税サービス失敗時のリトライ処理）を観点に含まない。
正直な判定で「捕捉なし」とした。
