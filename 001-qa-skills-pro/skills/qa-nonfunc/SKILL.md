# /qa-nonfunc — ISO/IEC 25010 非機能監査

仕様と実装を入力とし、ISO/IEC 25010の製品品質特性ごとに
「要求が存在するか」「実装が存在するか」「検証可能か」を根拠付きで棚卸しする。
機能が動くことしか見ないAI開発ループに、非機能の抜けを検出する層を足すのが役割。

## 手順

### 1. 共通ポリシーの読み込み

`../qa-shared/evidence-first-policy.md` と `../qa-shared/report-format.md` を読む。

### 2. プロファイルの決定

対象システムの種別（Web API / SPA・PWA / バッチ / CLI / モバイル）と利用文脈（社内 / BtoC / 規制産業）を特定し、
特性ごとの重み（高/中/低/対象外）を最初に宣言する。全特性を同じ深さで見ない。
重み付けの根拠（例: 「個人情報を扱うためセキュリティ=高」）を必ず書く。

### 3. 特性別監査

`references/iso25010-checklist.md` を読み込み、重み「高」「中」の特性について観点を走査する。
各観点の判定は3値:

- **実装あり** — コード位置を根拠に引用
- **実装なし・要求あり** — 仕様該当箇所を根拠に、欠落として指摘（severity付き）
- **要求も実装もなし** — 「要求の不在」自体を指摘するか対象外とするか、プロファイルに照らして判断し、判断根拠を書く

### 4. 出力

`../qa-shared/report-format.md` の指摘構造で出力する。レポート構成:

1. プロファイル宣言（種別・特性別の重みと根拠）
2. 特性別スコアカード（特性ごとに 0〜100 スコア＋主要根拠。重み付けを明示し、単純平均の総合点は出さない）
3. 指摘一覧（severity降順、category=25010特性名）
4. 監査範囲の宣言（対象外にした特性と理由、静的確認のみで実測していない旨）

## 制約（正直に宣言する）

- 本Skillはコード実行を伴わない静的監査。性能の実測値・負荷試験・ペネトレーションテストの代替にはならない。
  レポートに「静的監査であり実測を含まない」ことを必ず明記する。
- 実測が必要な観点は「実測推奨」として、測定方法（ツール・条件）を添えて出力する。
